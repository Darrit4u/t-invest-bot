# AGENTS.md

## Контекст проекта
- Проект: интрадей-бот по фьючерсам MOEX (`ES`, `SILVER`, `BRENT`, `NG`) на `5min`.
- Цель оптимизации: `winrate >= 75%` при минимум `1 закрытой сделке в день` для рабочей связки.
- Приоритет: реальная трейдинговая логика (структура цены, режим рынка, объёмы, сессии), а не только подгонка параметров.
- Практическое ограничение: прогоны длинные. Базовый тестовый диапазон: `2026-03-16` - `2026-04-10`.

## Обновление конфиг-архитектуры (актуально)
- Выполнен переход на настройку параметров по паре `инструмент+стратегия`.
- Набор параметров в конфиге сокращён до минимально необходимых для продакшена.
- Принцип: держать в конфиге только рычаги, реально влияющие на качество сигналов и риск; остальное фиксировать дефолтами в коде.

## Что сейчас в коде

### 1) `trend_pullback_vwap_ema`
Файл: `strategies/pullback_vwap_ema.py`

Как работает:
- Торгует только в `TREND`.
- Выбирает направление по `close vs vwap` и `ema_fast vs ema_slow`.
- Ищет импульс -> контролируемый pullback -> подтверждение.
- Опционально использует MTF-фильтр.

Что улучшено:
- Добавлены вспомогательные функции для расчёта session VWAP/EMA в логике setup.
- Добавлен time-filter по локальным часам (`blocked_entry_hours_local`).
- Добавлена поддержка фильтров по локальным дням недели (`blocked_entry_weekdays_local`, `allowed_entry_weekdays_local`).
- Добавлены опциональные фильтры качества подтверждения и proximity к якорной зоне.

Плюсы:
- Единственная стратегия, которая стабильно генерирует частоту сделок.
- Хорошо масштабируется под trade-management (TP1/TP2/BE).

Минусы/ограничения:
- На части инструментов частые стоп-кластеры.
- Чувствительна к часам торговой сессии и режиму волатильности.
- Без фильтра времени может давать много слабых входов.

---

### 2) `compression_breakout`
Файл: `strategies/compression_breakout.py`

Как работает:
- Торгует только в `COMPRESSION`.
- Ищет пробой диапазона и/или ретест после пробоя.

Что улучшено:
- Смягчён бинарный volume gate (сильное тело может компенсировать объём).
- Расширена логика ретеста (не только «очень большой» breakout).
- Добавлены relax-условия по EMA/VWAP в сильно overlap-режимах.

Плюсы:
- Логика рыночно корректная для импульсного выхода из боковика.

Минусы/ограничения:
- Очень редкий режим `COMPRESSION` на текущих данных.
- Часто нет достаточного подтверждения ретеста.
- В портфельных прогонах вклад в прибыль пока маленький.

---

### 3) `liquidity_sweep_reversal`
Файл: `strategies/liquidity_sweep.py`

Как работает:
- Торгует только в `BALANCE`.
- Ищет ложный вынос экстремума (sweep) и возврат в диапазон.

Что улучшено:
- Адаптивные границы `sweep_min/sweep_max` (ATR + range share + ticks).
- Адаптивный `return_close_distance`.
- Смягчены жёсткие local-balance ограничения.
- Добавлен MTF override для reversal-сценариев в балансе.

Плюсы:
- Идеологически полезна как mean-reversion слой.

Минусы/ограничения:
- На текущем интервале и активном профиле сигналы по-прежнему редки/нулевые.
- Очень чувствительна к микроструктуре инструмента и фильтрам режима.

## Пошаговый план для следующей нейросети (выполнять строго по порядку)

### Шаг 0. Ограничения прогона
1. Основной диапазон для всех сравнений: `--start 2026-03-16 --end 2026-04-10 --timeframe 5min`.
2. Если 3 запуска подряд слишком долгие, переключаться на `--instruments SILVER` и работать только по `SILVER`.
3. Не менять сразу много параметров: максимум 1-2 гипотезы за итерацию.

### Шаг 1. Базовая проверка перед правками
1. Прогон `active + trend_pullback_vwap_ema + SILVER`.
2. Зафиксировать: `signals/day`, `winrate`, `PF`, `net`, `max_drawdown`.
3. Использовать это как baseline итерации.

### Шаг 2. Итерации улучшения (приоритет)
1. Time filter:
   - анализировать winrate по часам и исключать локальные часы с системно плохим качеством.
2. Exit-логика:
   - тестировать только пару комбинаций `tp1_r/tp2_r` вокруг текущих значений.
3. Confirm quality:
   - аккуратно подбирать `confirmation_close_delta_atr` (без сильного снижения частоты).
4. Риск-контроль:
   - при необходимости добавить cooldown после серии SL (минимальная реализация).

### Шаг 3. Критерий принятия изменений
Изменение принимается только если одновременно:
1. `winrate` не ниже baseline и стремится к `>=75%`.
2. Частота не падает ниже `~1 сделки/день`.
3. `PF` не деградирует.
4. Нет падения unit-тестов (`python -m unittest discover -s tests -v`).

### Шаг 4. Работа с secondary-стратегиями
1. `compression_breakout` и `liquidity_sweep_reversal` включать только если они улучшают общий профиль, а не «для количества».
2. Если стратегия не даёт статистически полезного вклада, выключать её в боевом профиле.

### Шаг 5. Защита от переобучения
1. После каждой удачной правки прогонять дополнительный срез (например, последние 7-10 дней внутри диапазона) и сверять стабильность.
2. Не принимать решение по метрикам из 2-3 сделок.

### Шаг 6. Финализация
1. Обновить этот файл (`AGENTS.md`) новыми фактами.
2. Коротко записать: что изменено, какой KPI улучшился, какой компромисс по частоте.
3. Сохранить только необходимый набор параметров для редкой ручной поддержки на сервере.
